Course Objectives

  • Introduce Java and basic data structures.
  • Experience implementing solutions to problems with real-world implications.
  • Practice Java programming using files, threads, and sockets.
  • Understand core concepts including complexity, concurrency, and networking.
  • Experience working with an Integrated Development Environment.
  • Experience using a debugger.

Textbook

Grading Policy

Assignments 30%
Midterm 30%
Final 40%

The 'A+' grade will be considered only for the top 5% students in this course.

Assignments will be submitted via Canvas. It is the sole responsibility of the student to verify that submissions have been successfully uploaded prior to the due date. To verify a submission, revisit the submission page for the assignment to view your uploaded files.

Late Assignment Policy

Late submissions after the due date will NOT be accepted unless in cases of documented emergencies. As stated above, it is the sole responsibility of the student to verify that submissions have been successfully uploaded prior to the due date; incomplete submissions, for any reason, will not be accepted unless in documented emergencies.

Missed Exam Policy

Students must contact the instructor PRIOR to missing an exam to schedule a makeup exam. Otherwise, makeup exams will only be offered in cases of documented emergencies. Makeup exams will only be offered following the scheduled exam time. It is against the university policy to administer final exams before the posted date/time.

Plagiarism Policy

Plagiarism receives zero tolerance and can result in the automatic failure of this course. Discussion of course material and collaboration with other students is encouraged but each student must write/type and submit his/her own solution. Your code and proofs (if applicable) should never contain sections which are identical to the submission of another student, past or present. Submitted work must be entirely that of the student(s) whose name(s) appear(s) on the submission and not solicited in any way from others.

Any use of code from a source other than the course textbook or the examples from lectures/labs needs to be explicitly cited in the comments of your submitted code.

Submissions will be analyzed using Moss when applicable.

Message Board Policy

Students are encouraged to ask questions regarding the course, assignments, and/or exams on the course's message board. Posting on the message board can be done anonymously, if desired. Posts regarding outstanding assignment problems should be as high level as possible to not give other students an unfair advantage in solving the problem. Thus, code should only be posted when in reference to a specific error, which should be included in the post.

Tips for Success

Don't miss lectures. There are plenty of online resources for the course, but lectures provide a detailed introduction to new material and examples relevant to the current or upcoming assignments and exams. It is helpful to do the assigned reading prior to lectures so that you are familiar with the topics and terminology. Lectures also provide a forum for discussing the current course material. Also, this is why you're paying tuition.

Don't miss labs. Labs offer a supervised environment for hands-on programming. Labs are great for practicing your programming skills and/or working on the current assignment. Students may register and attend any lab session for the course provided that there are enough computers available.

Come to office hours. Office hours are your chance to get one-on-one help, ask questions, and seek general advising. Students may attend any GTF's or the instructor's office hours.

Do all of the assignments. Missing a single assignment will have a significant effect on your final grade. You will always get partial credit for partial solutions.

Think before you code. Every line of code has meaning and should move your program towards the solution. Arbitrarily adding lines of code in an attempt to obtain a solution may sometimes work, but does not scale well to more-difficult problems and can lead to code which is not elegant (i.e., has unnecessary lines of code) and not robust (i.e., does not handle all possible cases).

Allow enough time to work on programming assignments. You will get stuck somewhere on just about every programming problem. The issue is almost always something extremely minor, but will be time consuming to solve nonetheless. Part of learning to program is learning to overcome these obstacles. Sometimes it helps to take a break from coding or to switch to a different problem when you get stuck. If you still aren't making progress, ask for help by going to labs and office hours, emailing your GTFs, or posting on the message board.

Do as much of your assignment as possible before asking for help. Learning to write code is a difficult process which requires that you learn how to debug your code and how to use the available resources for the programming language(s). It is easy to convince yourself that you can write code if you're frequently getting help, but this tends to be problematic on exams.

Have fun!
